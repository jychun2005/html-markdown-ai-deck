2026.05 / AI Agent Workflow

HTML 是新的 Markdown？

Anthropic 工程師 Thariq Shihipar 提出：當 AI 產出的計畫、報告與審查內容變得又長又複雜，HTML 比 Markdown 更像一個能讓人持續參與的工作介面。

Core Argument

Markdown 是文件，HTML 可以是工作介面

  • 1AI agent 產出的內容越來越長，單純文字文件會降低人類審閱意願。
  • 2HTML 可以把同一批資訊變成可瀏覽、可篩選、可互動、可回傳的 artifact。
  • 3當人真的讀得下去、看得懂、能調整，監督品質才會提升。

Markdown

  • 線性閱讀
  • 適合筆記與版本控制
  • 長篇容易成為文字牆
  • 互動能力弱

HTML

  • 可視覺化布局
  • 可加入 SVG / CSS / JS
  • 可做互動決策介面
  • 分享後更像網頁工具

Why Now

早期瓶頸是 token；現在瓶頸是人會不會讀

Simon Willison 提到，GPT-4 早期 8K token 限制下，Markdown 的效率很值得。但大型模型與 coding agent 的輸出變成長規格、長計畫、長報告後，真正的問題變成：人類是否還願意審查？

過去輸出短，格式越省越好。
現在輸出長，審閱成本成為瓶頸。
下一步把輸出做成能操作的介面。

Capabilities

HTML 把 AI 輸出從報告推向互動物件

資訊密度表格、色彩、空間排列與視覺層級，讓同樣資訊更容易掃讀。
圖解能力SVG、流程圖、模組圖、資料流圖可直接嵌入一份檔案。
雙向互動滑桿、按鈕、拖拉、篩選、copy/export 讓回饋回到 agent loop。
分享成本HTML 上傳後就是網址，收件者不必理解 Markdown 檢視器。

Use Cases

最適合改成 HTML artifact 的場景

  • A規格、實作計畫、設計方向並排比較。
  • BPR review：diff、嚴重性、旁註與跳轉連結。
  • C研究報告、事故報告、週報與技術 explainer。
  • D設計系統、元件狀態、互動 prototype。
  • E臨時編輯器：ticket triage、feature flag、prompt tuner。

Critiques

反對聲音：HTML 不是免費午餐

  • 1同樣資訊，HTML 常比 Markdown 多出更多輸出 token。
  • 2生成速度較慢，且 verbose markup 可能稀釋模型注意力。
  • 3進版本控制時 diff 會很吵，不利長期共同維護。
  • 4HTML 適合看與互動，不一定適合當知識庫原始資料。

短期審閱

  • HTML 很強
  • 人更容易看懂
  • 適合一次性決策

長期記憶

  • Markdown 仍強
  • 可搜尋、可 diff
  • 更適合知識庫

Decision Rule

不是二選一，而是依任務選格式

用 HTML需要讓人讀完、比較、操作、調參、審閱、簡報、分享給非技術讀者。
用 Markdown需要長期保存、版本控制、grep、diff、筆記連結、作為原始知識庫。
混合策略Markdown 當 source of truth，HTML 當 review surface 或臨時 UI。
關鍵設計HTML artifact 最好提供 copy/export，讓互動結果能回到 agent 或 repo。

Implications

真正的變化：AI 文件開始像小工具

對教師、研究者、行政與產品團隊來說，HTML artifact 的價值不是炫技，而是把「請 AI 寫一份報告」升級成「請 AI 做一個我能操作的理解介面」。

教學把課程重點做成互動圖解、分層閱讀、即時小測。
研究把文獻關係、概念框架、比較表做成可展開的頁面。
行政把會議紀錄、待辦追蹤、風險清單做成可篩選 dashboard。
開發把規格、PR、架構圖做成可審閱 artifact。

Takeaway

格式之爭的核心，其實是人是否還在 loop 裡

Markdown 仍是優秀的長期記憶格式；HTML 則正在成為 AI agent 輸出給人審閱、理解、操作的中介層。未來的好工作流，很可能同時需要兩者。
